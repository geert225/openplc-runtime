# Runtime Synchronization Overhaul — Implementation Summary

Branches: `openplc-runtime: feat/sync-overhaul`, `strucpp: feat/threaded-var-sync`.
Validated on a Raspberry Pi 4 (aarch64) with a real Beckhoff **EK1818** EtherCAT
coupler and a 4-task stress program.

## What changed, step by step

**1. Remove dead synchronization (`c77e0b5`)**
Deleted `g_global_vars_mutex` and the entire `strucpp_set_locks` handoff — they
were created and passed to the `.so` but never locked by anyone (strucpp emits
no locking). `g_image_tables_mutex` stays; it's locked directly by the runtime.

**2. All plugin writes through the journal (`773eff2`)**
Migrated the EtherCAT bus thread's `%I` input writes from direct
`buffer_mutex`-guarded pokes to `journal_write_*` (s7comm and the Python plugins
already journaled their writes). Plugin writes are now race-free against the IEC
tasks.

**3. Lock-free journal (`0915260`)**
Replaced the single mutex-protected journal with a lock-free MPSC **double-buffer
flip**: producers claim a slot with one 32-bit `fetch_add` and publish with a
release store; the single consumer flips the active bank with one `exchange` and
drains the retired one. Needs only always-lock-free 32-bit + 8-bit atomics
(x86/arm64/armv7/riscv-with-A); compile-time **mutex fallback** otherwise.
Drop-on-overflow with logging; capacity 4096/bank.

**4. Threaded process-image model — the big lock is gone (`dda893e`, `80748b8`,
strucpp `feat/threaded-var-sync`)**
When the loaded `.so` was built threaded (exports `strucpp_threaded_abi`; the
runtime's `compile.sh` now passes `-DSTRUCPP_THREADED`), each IEC task runs its
body on **private storage with no global lock held**, so task bodies run in
parallel:
- **copy-in** (under the image mutex): drain the journal, then snapshot the
  task's located inputs from the runtime-owned image into program storage.
- **sync_in** (under a dedicated `global_mutex`): canonical globals → per-program
  private working copies (generated by strucpp).
- **run()** bodies: no lock held.
- **sync_out** (`global_mutex`): commit only *changed* globals back (memcmp vs a
  snapshot, so a task that only reads a shared global never clobbers a writer).
- **copy-out**: journal the task's *changed* located outputs (lock-free,
  dirty-diffed).

In threaded mode the image is no longer aliased to the `.so` located members
(the `temp_*` buffers back every slot), so the image stays private from program
storage and the runtime copies between them. strucpp gained
`ProgramBase::sync_in/sync_out/located_range` (appended vtable slots, no-op by
default) and `#ifdef STRUCPP_THREADED` per-program global local copies — so the
**same generated output still compiles and runs on an older, non-threaded
runtime** (it uses today's shared-reference + whole-body-lock path). A `.so`
without the capability flag uses the legacy path unchanged.

**Bonus: 32-bit Pi build fix (`075ec9f`)**
`std::atomic<uint64_t>` in the scan bookkeeping needs libatomic on armv6
(Raspberry Pi OS 32-bit). CMake now detects and links it only where required;
no-op on 64-bit, so the armv7 Docker / aarch64 builds are unaffected.

## Validation (Pi 4, aarch64, EK1818)
- Runtime + program compile clean; `.so` loads → `execution model: threaded
  process-image`.
- 4 IEC task threads (5/10/20/50 ms, distinct priorities) + the EtherCAT bus
  thread run **in parallel** — CPU time accrues proportional to each task's rate,
  confirming no serializing lock.
- ~3 min sustained run under multi-task + live EtherCAT I/O: **zero
  errors/crashes/overflow**, journal healthy.
- Lock-free journal path selected and exercised by real EK1818 input traffic.

## No dead mutex
`g_image_tables_mutex`, `g_global_mutex` (threaded global sync), `state_mutex`,
`plc_tasks_lock`, `python_blocks_mutex`, `log_mutex`, `buffer_mutex` (plugin
reads via the `mutex_take` thunk) are all live; the journal's mutex exists only
in the non-lock-free fallback. None dead.

## Known remaining refinements (not regressions; documented)
- **Plugin output (`%Q`) reads still use `buffer_mutex`**, a different lock than
  the journal drain's image mutex, so a *multi-byte* `%Q` read by a plugin can
  tear against a concurrent drain (the pre-existing race, unchanged; EK1818's
  bit outputs are byte-atomic and unaffected). Migrating plugin reads to a brief
  image-lock read accessor would close this and retire `buffer_mutex`.
- **Debug / OPC-UA path** (reads/writes any variable) is intentionally untouched
  this pass and still takes no lock.
- **Compound (struct/array/string) globals**: `sync_out` diffs via `memcmp` on
  the raw value (correct), but only scalar/located commit through the typed
  journal; large compound globals would want a bulk-copy commit path.
- The IEC `PRIORITY` value maps 1:1 to SCHED_FIFO (higher = higher); confirm this
  matches the editor's intended convention.
